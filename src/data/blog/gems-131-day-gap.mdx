วันหนึ่งเปิดดูข้อมูล GEMS (ดาวเทียมวัด aerosol) ปี 2025 แล้วเจอช่องโหว่ — ตั้งแต่ 20 กุมภาพันธ์ ถึง 30 มิถุนายน **131 วันรวด มีไฟล์ 0 ไฟล์** สมมติฐานแรกที่ทุกคนคิดคือดาวเทียมดับ หรือ instrument downtime

แต่หลักการของงานนี้คือ **ดูที่ pattern ไม่ใช่ที่ความตั้งใจ** — ก่อนจะสรุปว่าดาวเทียมเสีย ต้องพิสูจน์ก่อน

## ขั้นแรก: ข้อมูลหาย หรือ แค่โหลดไม่ลง?

ไปเช็คที่ต้นทาง — NESC API ที่เป็นแหล่งข้อมูล GEMS กลับมีไฟล์ครบทุกวันในช่วงนั้น ไม่มีช่วงไหนขาด แปลว่า **ดาวเทียมไม่ได้เสีย ข้อมูลอยู่ครบ** ที่หายคือฝั่งเราโหลดไม่ลงเอง

ตรงนี้คือจุดที่ต้องระวัง: gap ในไฟล์บนดิสก์ ไม่ได้แปลว่าข้อมูลไม่มี มันอาจแปลว่า pipeline พัง พอแยกสองอย่างนี้ออกจากกันได้ ปัญหาก็เปลี่ยนจาก "ข้อมูลสูญหาย" เป็น "หาบั๊กในตัวดาวน์โหลด"

## ขั้นสอง: ทำไมมันค้างที่ 20 ก.พ. เป๊ะ

ตัวดาวน์โหลดรันใต้ PM2 ตั้ง `cron_restart` ให้ restart ทุกชั่วโมง ไอเดียคือกัน process ค้าง แต่พอ restart PM2 ส่ง **SIGINT** เข้า process Python

แล้ว Python แปลง SIGINT เป็น `KeyboardInterrupt` ซึ่ง**ไม่ได้สืบทอดจาก `Exception`** — มันอยู่ใต้ `BaseException` โค้ดดักไว้แบบนี้:

```python
try:
    download_chunk(date)        # ติดอยู่ตรง SSL read พอดี
except Exception as e:          # จับ KeyboardInterrupt ไม่ได้!
    log.error(f"retry {date}: {e}")
    retry(date)
```

พอ SIGINT มาตอนกำลัง SSL read `except Exception` เลยจับไม่ติด process ตายทั้งยืน PM2 เห็นตาย ก็ restart ใหม่ — **แล้วเริ่มนับจาก 1 มกราคมใหม่ทุกครั้ง**

วนแบบนี้ ~130 รอบ แต่ละรอบโหลดได้ถึงแถวๆ 20 ก.พ. ก็โดน SIGINT ตัดจบพอดี เลยค้างอยู่ตรงนั้นไม่ไปไหน gap 131 วันจึงไม่ใช่ดาวเทียมดับ แต่คือ cron restart ที่ฆ่าตัวเองทุกชั่วโมง

## ขั้นสาม: กู้คืน โดยไม่แกล้งว่าไม่เคยพัง

แก้ root cause สองชั้น — เปลี่ยนตัวจับให้ครอบ `BaseException` (หรือ handle SIGINT ตรงๆ ให้ save state ก่อนตาย) แล้วให้ดาวน์โหลด resume จากวันล่าสุดที่มี ไม่ใช่เริ่มจาก 1 ม.ค.

ส่วนข้อมูลที่หาย ก็ยิง backfill ย้อนเป็น chunk ละ 10 วัน (14 chunks) คู่กับ Airflow DAG `gems_backfill_2025` ที่ trigger ซ้ำได้ ผลคือเดือน ก.พ.–มิ.ย. เติมกลับครบ

ระหว่างขุดยังเจอของแถม — `air4thai-cron` (ตัวดึงข้อมูล BAM อ้างอิงจาก PCD) ถูก stop เงียบๆ ไป 3 รอบ ถ้าปล่อยไว้ข้อมูลอ้างอิงก็หายอีกชุด เลย restart กลับมาด้วย

## สามบทเรียนที่เอาไปใช้ต่อได้

- **gap ในข้อมูล ไม่เท่ากับ ข้อมูลที่ไม่มี** — เช็คต้นทางก่อนเสมอ ก่อนสรุปว่าเซ็นเซอร์หรือดาวเทียมเสีย หลายครั้งของอยู่ครบ แค่ pipeline พัง
- **`except Exception` ไม่จับ `KeyboardInterrupt`/SIGINT** — สองตัวนี้อยู่ใต้ `BaseException` ถ้าระบบโดน signal สั่งปิด โค้ด retry ที่ดักแค่ `Exception` จะปกป้องอะไรไม่ได้เลย
- **restart ต้อง resume ไม่ใช่เริ่มใหม่** — cron restart ที่รีเซ็ตงานกลับไปจุดเริ่มทุกครั้ง คือเครื่องผลิต gap ชั้นดี

131 วันนั้นไม่ได้ถูกลบทิ้งให้ชุดข้อมูลดูสะอาด แต่ถูกบันทึกว่าหายเพราะอะไร แล้วเติมกลับพร้อมหลักฐาน — เพราะ gap ที่อธิบายได้ มีค่ามากกว่าข้อมูลที่ดูครบแต่ไม่รู้ที่มา
