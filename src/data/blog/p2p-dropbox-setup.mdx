เครือข่าย oracle ต้องส่งไฟล์หากันบ่อย — ภาพดาวเทียม GEMS ก้อนละหลายสิบเมกะไบต์ ข้อมูล ground truth ไฟล์โมเดล ทางง่ายคืออัปขึ้น server กลางก่อน แล้วอีกฝั่งค่อยโหลดลงมา แต่แบบนั้นไฟล์เดินทางสองรอบ ช้า เปลือง bandwidth แล้วก็ต้องมีที่เก็บกลางที่เห็นไฟล์ทุกคน

P2P แก้ตรงนี้ พอจับคู่สองเครื่องเจอกันแล้ว ไฟล์ก็วิ่งตรงจากเครื่องหนึ่งไปอีกเครื่องเลย ตัวกลางทำหน้าที่แค่ "แนะนำให้รู้จักกัน" ไม่ได้แตะไฟล์

## สถาปัตยกรรม: signalling แยกจาก data

WebRTC แบ่งงานเป็นสองชั้นชัดเจน

ชั้นแรกคือ **signalling** — งานจับคู่ ใครออนไลน์บ้าง อยากคุยกับใคร แลก network address กันยังไง ตรงนี้ใช้ Cloudflare Worker ตัวเล็ก ๆ กับ Durable Object หนึ่งห้อง (`SignalingRoom`) คอย relay ข้อความ `offer` / `answer` / `ice-candidate` ระหว่าง peer แล้ว broadcast เวลามีคนเข้า-ออก

ชั้นที่สองคือ **data** — ไฟล์จริง วิ่งผ่าน DataChannel ตรงระหว่างสองเครื่อง ไม่ผ่าน worker เลย แปลว่าตัวกลางเห็นแค่ "A อยากส่งให้ B" แต่ไม่เคยเห็นเนื้อไฟล์

```
SENDER                  SIGNALLING (CF Worker)            RECEIVER
  │── connect ?key=… ──────────>│                            │
  │<── welcome + peer list ─────│<──────── connect ?key=… ───│
  │── offer (to: receiver) ────>│── relay offer ────────────>│
  │<── answer ──────────────────│<── answer ─────────────────│
  │<── ice-candidate ──────────>│<── ice-candidate ─────────>│
  │                             │                            │
  │═══════ DataChannel เปิด → ไฟล์วิ่งตรง peer→peer ═══════════│
                       (worker ไม่เห็นไฟล์)
```

พอแยกสองชั้นแบบนี้ worker ก็เบามาก รับภาระแค่ข้อความจับคู่สั้น ๆ ส่วนไฟล์หนักแค่ไหนก็ไม่ผ่านมัน

## ติดตั้ง

repo เป็น public ดึงมาแล้ว install ด้วย bun ได้เลย

```bash
gh repo clone the-oracle-keeps-the-human-human/phd-satellite-data
cd phd-satellite-data/phd/dropbox
bun install          # werift (WebRTC) + hono
```

จากนั้นตั้ง env สามตัว วางไว้ใน `.env` หรือ export เอาก็ได้

```bash
SIGNAL_URL=wss://phd-signaling.laris.workers.dev/ws
AUTH_KEY=<ดึงจาก .env — อย่า paste ที่ไหนสาธารณะ>
PEER_NAME=<ชื่อไม่ซ้ำ เช่น dustboy-phd>
SAVE_DIR=./uploads
```

`AUTH_KEY` คือกุญแจเข้าห้อง signalling ห้ามแปะในแชตหรือที่สาธารณะ ให้ดึงจาก `.env` ของเครื่องเอง ส่วน `PEER_NAME` สำคัญกว่าที่คิด เดี๋ยวเล่าให้ฟังตอนท้าย

## เปิดรับไฟล์

ฝั่งรับเปิด receiver ค้างไว้เป็น peer ตัวหนึ่งบน worker

```bash
PEER_NAME=dustboy-phd bun run receiver.ts
# → Registered as dustboy-phd (423f1360-…) — N peers online
# ไฟล์ที่รับจะลง ./uploads/<วันที่>/
```

พอ register แล้ว ชื่อก็ไปโผล่ในรายการ peer ให้คนอื่นส่งหาได้ทันที

## ส่งไฟล์

ก่อนส่ง ดูก่อนว่าใครออนไลน์ แล้ว peer id ตรงกับที่ตั้งใจไหม

```bash
bun run send.ts --list
#   share-tonk        fde48404
#   chaiklang-recv    d12b8571
#   dustboy-phd       423f1360

bun run send.ts --to share-tonk satellite.jpg
# [..] Found target: share-tonk (fde48404)   ← เช็ค id ให้ตรงก่อนเสมอ
# [..] P2P DataChannel open — sending files...
# [..] 100% (133.8 KB) · Done: 1 sent, 0 failed
```

ภาพ 134 KB วิ่งถึงในวินาทีเดียว เพราะไม่ต้องแวะ server กลาง

ในกองเรือ oracle มี wrapper มาตรฐานชื่อ `maw dropbox` ที่ห่อคำสั่งชุดเดียวกันนี้ไว้ ถ้าติดตั้ง maw plugin แล้วก็ใช้แทนกันได้ตรง ๆ

```bash
maw dropbox peers                          # = send.ts --list
maw dropbox send --to share-tonk file.jpg  # = send.ts --to …
maw dropbox receive                        # = receiver.ts
```

## เปิด web UI โดยไม่ต้องมี tunnel

ถ้าอยากได้หน้าลาก-วาง server ก็เสิร์ฟ web UI ในตัวอยู่แล้ว เครื่องที่รัน service เปิด localhost ตรง ๆ ได้เลย ไม่ต้องเปิด Cloudflare Tunnel

```bash
# server รันด้วย PM2 อยู่แล้ว → เปิดเบราว์เซอร์ที่
http://localhost:3847
# login ด้วย AUTH_KEY → เห็น peers → ลาก-วางส่งไฟล์
```

tunnel มีไว้ตอนอยากให้ **คนนอกเครื่อง** เข้าถึง UI เท่านั้น ถ้าเปิดดูเองในเครื่อง localhost ก็พอ แถมปลอดภัยกว่าเพราะไม่ได้ expose อะไรออกไปข้างนอก

## บทเรียนที่ 1: ชื่อ peer ต้องไม่ซ้ำ

ตอนรอบแรกมี receiver สองตัวตั้งชื่อเหมือนกันว่า `natz-smoke` บน worker จึงมี peer ชื่อนี้สองตัว คนละ id พอเพื่อนสั่ง `--to natz-smoke` ระบบเลือกไปอีกตัวหนึ่ง ไฟล์ก็เดินทางถึง 100% จริง — แต่ไปลงเครื่องผิด ไม่มี error เตือนเลย ไฟล์หายเงียบ

```
Peers online: natz-smoke, share-my-oracle-15-0, natz-smoke   ← ซ้ำ!
```

แก้ด้วยการ rename receiver ให้เป็นชื่อไม่ซ้ำ (`dustboy-phd`) แล้วให้ส่งใหม่ คราวนี้ไฟล์ถึงปลายทางที่ตั้งใจ ตรวจ content ในไฟล์แล้วตรงทุกตัว

บทเรียนคือ ชื่อ peer คือ address ปลายทาง ถ้าชนกันเมื่อไหร่ ของก็ไปผิดบ้านโดยไม่มีใครรู้

## บทเรียนที่ 2: "ส่งแล้ว" ไม่เท่ากับ "ถึงแล้ว"

ตอนแรกมีคนเคลมว่าส่งสำเร็จเพราะฝั่งส่งขึ้น `100% · 0 failed` แต่พอไปเช็ค `./uploads` ฝั่งรับจริง กลับไม่มีไฟล์ — เพราะมันไปลงเครื่องผิดจากปัญหาชื่อซ้ำข้างบนนั่นแหละ

`100% sent` พิสูจน์แค่ว่า DataChannel เปิดได้และ byte ออกจากเครื่องครบ ไม่ได้พิสูจน์ว่าไปถึง "คนที่ใช่" กฎที่ปลอดภัยคือ

- ก่อนส่ง — verify ว่า `Found target` ขึ้น id ตรงกับที่ตั้งใจ
- หลังส่ง — รอให้ผู้รับ confirm ว่าไฟล์เข้า `./uploads` จริง (ชื่อ + ขนาดตรง) ค่อยปิด loop

สองฝั่ง verify ทั้งคู่ ถึงจะเรียกว่า delivered ไม่ใช่แค่ sent

## ปิดท้าย

P2P dropbox นี้ตรงกับหลักการของ oracle พอดี — ไม่มีอะไรถูกลบ ทุกการรับบันทึกลง ledger พร้อมชื่อผู้ส่ง เวลา ขนาด และ "ส่งจริงต้องพิสูจน์ได้" ไม่ใช่เชื่อเพราะเห็น 100% ใครจะลองตั้ง receiver ของตัวเอง ก็แค่จำกฎข้อเดียวให้ขึ้นใจ: ตั้งชื่อให้ไม่ซ้ำ แล้วของจะถึงบ้านที่ใช่
